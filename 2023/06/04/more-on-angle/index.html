EXT or not?

这个其实是之前用Vulkan Backend就发现的问题: 打开RenderDoc后会优先使用非EXT版本的API，导致glCopyImageSubDataEXT调用被劫持成glCopyImageSubData。
我之前给RenderDoc提过split code path for glCopyImageSubDataEXT&glCopyImageSubData，但是作者的回复还是比较有道理的——在GLES 3.1 Context的情况下，ANGLE其实就不应该返回glCopyImageSubData这个接口(虽然里面是空实现)。

临时解决方案是ANGLE里ValidateCopyImageSubData里的ES 3.2限制降到3.1。

glProgramBinary

偶然间关掉RenderDoc之后发现启动时大概率会闪退的一个问题，看了下堆栈是崩在了glProgramBinary中

看了下这块代码，然后在ANGLE的test case里用现有的ProgramBinaryUniformResetCase改了改，发现能稳定复现了这个问题: 本质是es 3.1的uniform sampler binding point压根没考虑到…
给官方提了glProgramBinary crash with unifor binding point，回复的非常快但是吐槽无力…两年前就有人提过这个事情还没修(难怪我搜glProgramBinary没找到这个issue…)

本来想直接禁用shader binary cache绕开，后来发现glLinkProgram的时候ANGLE内部也会尝试复用Program，导致一样的闪退。

行吧，同样是非常临时的方案糊弄了下，目前跑起来没啥问题倒是…

RenderDoc Replay

这个其实不是ANGLE的问题了，而是RenderDoc的BUG…
在分帧分块调用glCopyImageSubDataEXT时，RenderDoc只能保存当前帧的数据，导致目标Texture其它部分的历史数据会直接丢失掉。

这个问题我其实当年提supporting full glCopyImageSubData的时候也发现了，只不过后面一直没空去填坑…这次准备一起搞一下orz
